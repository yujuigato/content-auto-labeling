Background

這個系統在解決什麼問題？

內容團隊每天產出大量文章，手動標籤耗時且不一致；業務需要依場景、議題找到對應的知識內容，卻找不到。

😣 過去的痛點

  • 文章標籤全靠人工，速度慢且標準不統一
  • 無法快速按「場景」或「議題」篩選相關文章
  • 標籤體系更新後，舊文章需要重新審閱
  • 產品標籤需要比對同義詞，純 LLM 容易出錯

✅ 系統如何解決

  • LLM 判斷場景與議題（語意理解），關鍵字比對產品（精確匹配）
  • 三層標籤結構（大類 > 小類），搭配同義詞字典
  • FastAPI 提供即時 API，單篇或批次皆可
  • 標籤結果自動同步至 SQL Server 知識庫
Architecture

混合式標籤引擎

LLM 擅長語意理解，關鍵字擅長精確比對，兩者各司其職，互補不足。

flowchart TD A["📄 文章\n(MySQL)"] --> B["HTML 清洗\nBeautifulSoup"] B --> C["文章內容"] C --> D["🤖 Gemini LLM\n場景 × 議題 判斷"] C --> E["🔑 關鍵字比對\n同義詞字典"] D --> F["場景標籤\n議題標籤"] E --> G["產品標籤"] F & G --> H["三層標籤結果\n場景 × 議題 × 產品"] H --> I["FastAPI 回傳"] H --> J["SQL Server 同步\nTGAJ / TGAK / TGAD"] style D fill:#0891b2,color:#fff style H fill:#059669,color:#fff

為什麼是混合式，不是純 LLM？

產品標籤需要精確比對產品代碼（TT / T100 / CM / WF / SM），LLM 的創意性回答容易引入不存在的標籤。 關鍵字比對搭配同義詞字典可以達到 100% 的標籤合法性；場景與議題則需要語意理解，這正是 LLM 的強項。 兩者分工，各取所長。

API Design

FastAPI 即時分類服務

端點功能輸入輸出
POST /label單篇文章標籤文章 HTML / 純文字三層標籤 JSON
POST /batch批次標籤文章 ID 清單批次標籤結果
POST /sync同步知識庫標籤結果寫入 TGAJ/TGAK/TGAD
GET /dict查詢標籤字典—當前有效標籤清單
