Background · System 1

這個系統在解決什麼問題？

內容團隊大量產出文章，但沒有標籤就無法讓業務依場景快速找到對應的知識內容；手動標籤速度慢且標準不一。

😣 過去的痛點

  • 文章標籤全靠人工，一篇要花 10–15 分鐘
  • 不同人打的標籤標準不一，搜尋結果不準確
  • 標籤體系更新後，舊文章需要全部重新審閱
  • 產品標籤需要比對同義詞，純 LLM 容易產生不存在的標籤

✅ 系統如何解決

  • LLM 負責語意理解（場景、議題），關鍵字負責精確比對（產品）
  • 三層標籤結構搭配同義詞字典，確保標籤合法性 100%
  • FastAPI 提供即時 REST API，批次重新標籤只需分鐘
  • 標籤結果自動同步至 SQL Server 知識庫
How It Works

系統如何運作？

為什麼是「混合式」，不是純 LLM？

產品標籤需要精確比對內部產品代碼（不能有 LLM 自創的標籤），用關鍵字 + 同義詞字典可以確保 100% 合法性。場景與議題需要語意理解，這正是 LLM 的強項。兩者分工，各取所長。

1

擷取與清洗

從 MySQL 讀取文章，BeautifulSoup 清洗 HTML 標籤，取出可分析的純文字。

處理各種 HTML 格式差異，包含圖片說明文字、表格、引用區塊的清洗邏輯
2

Gemini LLM 語意判斷（場景、議題）

LLM 理解文章主旨，從預定義的標籤體系中選出最符合的場景與議題標籤。

Prompt 包含完整標籤定義與範例，引導 LLM 只從合法標籤中選擇
3

關鍵字比對（產品）

比對同義詞字典，識別文章中提到的產品，輸出對應產品代碼。

同義詞字典由 Excel 維護，系統啟動時自動載入，更新字典不需要改程式
4

結果輸出與同步

三層標籤彙整後，透過 FastAPI 回傳，同時自動寫入 SQL Server 知識庫。

Engineering Challenges

解決了哪些技術難題？

Challenge 1

確保 LLM 不會產生不存在的標籤

LLM 有創意性，可能回傳系統中不存在的標籤（幻覺問題），導致標籤無效。

↓ 解決方案

Prompt 中放入完整標籤清單 + 輸出格式限制；後處理驗證每個標籤是否在合法清單中，不合法則丟棄

Challenge 2

HTML 清洗的複雜性

文章來源格式不一，有的用 div、有的用 p、有的有廣告嵌入，直接送給 LLM 雜訊太多。

↓ 解決方案

BeautifulSoup 針對不同文章格式分別處理，保留段落結構的同時去除導覽、廣告、作者欄位等無關內容

Challenge 3

標籤體系版本更新

業務場景隨市場變化，標籤定義需要定期調整，但每次更新都要重新處理所有舊文章。

↓ 解決方案

FastAPI 批次端點 + 版本標記：只重新標籤版本號低於當前版本的文章，避免全量重跑

Challenge 4

產品同義詞維護

同一個產品可能有品牌名、縮寫、舊名稱多種叫法，靠人工記住所有變體不可行。

↓ 解決方案

同義詞字典集中在 Excel 維護，非技術人員可直接更新；API 啟動時自動載入，更新立即生效

系統二：法人產行業標籤
Background · System 2

法人產行業標籤系統在解決什麼問題？

企業資料散落在 104、商業司、工廠登記、財稅等四個政府與商業資料庫，每個來源的行業分類系統都不一樣，無法直接比較或合併。

😣 過去的痛點

  • 同一家公司，四個來源給的行業代號完全不同，不知道哪個準
  • 大中小細類層級複雜，人工對照費時且容易出錯
  • 公司名稱含「製造」卻被分在服務業的錯誤分類持續累積
  • 新增法人時需要人工查表判定行業，無法規模化

✅ 系統如何解決

  • 整合四大來源，建立統一的行業代號判定流程
  • 關鍵字規則優先：製造/組裝→C，流通/批發→F，醫院→M8110
  • 規則覆蓋不到的案例，依來源優先序自動選擇最可信的代號
  • 最終回退到語意模型，確保每家公司都有有效的分類

💡 什麼是產行業代號？

台灣行業標準分類採四層結構：大類（英文字母）→ 中類（2碼）→ 小類（3碼）→ 細類（4碼）。例如：
C → C10 → C101 → C1011（食品製造業 → 屠宰及肉品加工）
業務可依行業代號快速找出同類型客戶，複製成功銷售策略。

1

四大來源資料彙整

從 104、商業司、工廠登記、財稅資料庫各自取得公司的行業描述或代號，對應到統一的行業大中小細類表。

每個來源都有獨立的對照表（104對照表、商業司對照表、工廠對照表、財稅對照表），統一轉換為標準行業代號
2

直接命中判定

若公司自填的行業類別已完全符合行業定義，直接採用，不需進入後續判斷流程，速度最快且最準確。

3

關鍵字規則判定

依產品說明或公司名稱的關鍵字，套用規則直接判定行業大類。

製造 / 組裝 / 研發 → C 製造業
流通 / 買賣 / 批發 / 零售 / 貿易 → F 批發及零售業
無店面 / 電商 → F48 無店面零售業
公會 / 協會 / 基金會 → O94 職業及類似組織
醫院 → M8110・診所 → M8140・長照機構 → M8130
4

來源優先序仲裁

若規則無法直接判定，且多個來源給出不同結果，依可信度優先序自動選擇最合適的來源。

商業司（公司正式登記）> 財稅（申報資料）> 104（公開求才）> 工廠登記
5

語意模型回退

仍有疑義時，以語意模型分析公司描述文字，輸出最符合的行業代號，確保所有法人都有有效分類。

Engineering Challenges · System 2

產行業系統的技術難題

Challenge 1

四個來源的行業碼體系完全不同

104 用自訂職能分類、商業司用公司法行業、財稅用稅務行業碼、工廠用工廠行業碼，四套體系互不相通。

↓ 解決方案

為每個來源建立獨立對照表，統一映射到標準行業分類（主計處第 11 版）；對照表以 Excel 維護，可持續更新

Challenge 2

關鍵字規則的邊界案例

「國際貿易」公司的主要產品是「組裝服務」，關鍵字同時命中 C（製造）和 F（流通），產生衝突。

↓ 解決方案

規則設計加入優先序：產品說明關鍵字優先於公司名稱關鍵字；細類規則優先於大類規則；無店面/電商優先於一般零售

Challenge 3

行業層級的部分匹配問題

某來源只提供大類（C），另一來源提供細類（C1011）。直接比較會誤判為衝突，其實大類是相容的。

↓ 解決方案

比對時先對齊到相同層級再判斷一致性；採用細類的來源優先於只有大類的來源，資訊越完整越可信

Challenge 4

行業定義隨法規更新

主計處行業分類定期修訂（目前第 11 版），舊代號可能被合併、拆分或更名，歷史資料需要維護。

↓ 解決方案

對照表版本化管理，新舊代號保留映射關係；定期比對各來源更新情況，確保對照表與最新版本同步
